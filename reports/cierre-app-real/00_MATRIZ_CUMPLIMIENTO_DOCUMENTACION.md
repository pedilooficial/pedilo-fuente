## Criterios aplicados

- No se considera cierre por existencia de archivo ni por tests solamente.
- Si la documentación exige uso real y la app tiene algo visual/no persistente, se marca `parcial` o `no cumple`.
- Si UI y backend no están conectados en ambos sentidos, se marca `parcial` o `no cumple`.
- Si backend existe pero no puede usarse desde UI real, se marca `parcial`.
- Si una capacidad está preparada pero no habilitada, sólo cumple si la UI lo comunica de forma clara.
- Pantallas en desarrollo, shell o placeholders no cuentan como uso real final.

## Principales faltantes críticos

1. UI visible por rol no aprobada para uso real, con pantallas shell/parciales que deben implementarse o bloquearse claramente.
2. Ejes completos de Pedido Vivo: financiero, comunicación, archivo, timeouts/fallbacks ejecutables y snapshots completos.
3. Finanzas/cobro/cierre de caja: no hay cierre real suficiente para operar dinero con repartidores.
4. Backend/rules pendientes para módulos nuevos: config, roles CRUD, pagos, cierre, modos, chat/notificaciones y cancelación pública.
5. Hardening real en celulares y carga no ejecutado como certificación de cierre.

## Principales faltantes altos

1. Configuración Admin persistente y versionada.
2. Alta/baja/gestión real de roles desde Admin.
3. Cancelación pública por estado.
4. Catálogo administrable, variantes, extras y stock.
5. Modos lluvia/saturación/mantenimiento/emergencia.
6. Gestión Admin de reclamos, incidencias completas y producto no disponible.
7. WhatsApp/cola/fallback, chat interno y notificaciones.
8. Dashboards por rol y salud/métricas visibles para Admin.
